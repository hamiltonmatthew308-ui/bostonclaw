# IM 自动化接入（飞书 / 企微）

目标：让员工用 IM（飞书/企微）“对话驱动” OpenClaw / FreeClaw / 其他 runtime，同时满足企业网络与成本约束。

## 为什么“纯本地 FreeClaw”很难直接接 IM

FreeClaw/OpenClaw 在本机启动后，Gateway 通常监听 `127.0.0.1:<port>`（本地回环地址）。

IM 平台（飞书/企微）的 Bot/事件回调通常要求：
- 一个**平台服务器可访问**的 HTTPS 回调地址（公网或企业可达）
- 稳定的长连接/回调链路（TLS、证书、域名）

因此“本机-only”天然不可达：平台服务器无法访问员工电脑的 `127.0.0.1` 或内网 IP。

## 推荐架构（企业共享服务器：中继/网关）

这是最适合“1500 人规模”落地的路径：只让 IM 平台连一台（或一组）公司服务器。

**组件**
- `IM Gateway / Relay`（共享服务器，公网可达）
  - 接收飞书/企微事件回调（HTTPS）
  - 做签名校验、权限、审计、限流
  - 把消息路由到后端 OpenClaw Gateway（本地/云端）
- `Worker Agent`（员工本机或部门机器，内网）
  - 主动向 Relay 建立出站连接（WebSocket/长连接）
  - 接收任务并在本机执行（如果需要“本地文件/内网系统”能力）

**关键优势**
- 不需要每台电脑做内网穿透
- 统一审计与权限控制
- 并发扩展清晰（Relay 可水平扩展）

**Lobster 的产品落点**
- 安装器问答中增加：“需要用飞书/企微聊天驱动吗？”
  - 选“是”：引导到“共享服务器接入”流程（填 AppID/Secret/回调 URL，或跳转到 IT 配置文档）
  - 选“否”：继续本地模式（dashboard/CLI）

## 实验选项（个人隧道 / 内网穿透）

适合个人测试或小范围演示，不建议全员规模化。

做法：在员工电脑上把本地 Gateway 暴露成公网 HTTPS，再把回调地址填到飞书/企微。

可选工具（示例）
- Cloudflare Tunnel（cloudflared）
- ngrok
- frp（需要自建服务器）

风险
- 公司安全策略可能禁止
- 网络波动导致回调不稳定
- 需要管理证书/域名/回调变更

## 现实边界与结论

1. “本地-only + IM 控制”必须有“平台可达的入口”：要么上云入口，要么隧道，要么企业网关。
2. 对企业规模落地：默认走“共享服务器 Relay”方案；隧道只作为实验路径说明。
3. 对比赛 Demo：至少跑通一个闭环（建议：共享服务器最小版），并在汇报中明确对比三种方案的成本与可控性。
