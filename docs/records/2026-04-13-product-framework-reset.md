# Lobster 项目框架重置文档

- 日期：2026-04-13
- 记录类型：产品框架重置 / 实施基线
- 适用范围：`/Users/jouska/Projects/lobster-community`

## 1. 背景

Lobster 项目最早的方向是围绕 OpenClaw 社区、Agent 协作网络和可视化运营站点展开。但在进一步梳理后，当前阶段最核心的问题并不是“如何做一个更炫的 Agent 社区”，而是：

1. 员工不知道该装什么。
2. 员工不会安装和配置现有 AI Agent Runtime。
3. 员工装完之后，不知道立即能拿来做什么。
4. 公司在推进 AI 采用时，需要一个成本可控、可扩展、可讲清楚的落地方案。

因此，项目需要从“OpenClaw 社区平台”重置为“企业 AI Agent 的安装与模板平台”。

## 2. 新的一句话定位

Lobster 是一个企业内部 AI Agent 安装桥与模板中心，帮助员工按场景快速选择合适的 Runtime，自动完成安装配置，并一键获得可直接投入工作的智能体模板。

## 3. 当前阶段要解决的问题

本期优先解决的是 adoption friction，而不是模型能力上限。

具体来说，Lobster 负责解决：

1. 安装门槛高
2. 配置门槛高
3. 模板复用弱
4. 渠道接入散
5. 公司内部 AI 推广缺少统一入口

## 4. 产品边界

### 4.1 我们要做什么

1. 做一个安装桥
2. 做一个模板中心
3. 做 Runtime 适配层
4. 做基础渠道接入能力
5. 做一个轻量 Web 入口承载模板浏览、产品说明和后续运营能力

### 4.2 我们暂时不做什么

1. 不自研新的 AI Runtime
2. 不自研大模型推理服务
3. 不做人手一套服务器
4. 不把 Agent-to-Agent 社区协作作为一期主线
5. 不把本地模型作为主部署路径

这些能力未来可以扩展，但不能成为当前施工期的主目标。

## 5. 核心产品结构

Lobster 当前建议收敛为四层：

### 5.1 安装桥

目标：把复杂文档、CLI 指令和环境配置收敛为向导式流程。

核心能力：

1. 根据场景推荐 Runtime
2. 选择本地安装或云端部署路径
3. 自动执行安装脚本、依赖检测和初始化
4. 自动生成或写入配置文件

安装桥不是新的 Runtime，而是现有 Runtime 的 adoption accelerator。

### 5.2 模板中心

目标：让员工不是从空白输入框开始，而是从可执行的角色模板开始。

模板不是 Prompt 文本库，而是配置包。

一个模板包至少包含：

1. 模板名称与适用人群
2. 角色定义
3. 系统提示词
4. 推荐模型与成本等级
5. 工具配置
6. 渠道绑定建议
7. 对应 Runtime 的配置映射
8. 输出格式约束

### 5.3 Runtime 适配层

目标：不把项目成败绑定到某一个 Runtime。

当前候选适配对象：

1. OpenClaw
2. Accomplish
3. Hermès Agent
4. 后续其他兼容方案

适配层负责把模板配置翻译为目标 Runtime 可消费的配置文件、命令或安装动作。

### 5.4 轻量 Web 入口

目标：先承载模板浏览、方案说明和后续运营扩展，不在一期承载复杂业务。

一期建议承载：

1. 产品定位说明
2. 模板列表与筛选
3. 安装路径说明
4. 渠道接入说明
5. 后续统计能力预留

## 6. 一期产品范围

一期只做最能形成闭环的部分。

### 6.1 一期必须完成

1. 安装桥基础框架
2. Runtime 选择流程
3. 模板中心的数据结构
4. 模板一键写入配置
5. 至少一个本地 Runtime 适配
6. 至少一个远程渠道接入方案的占位设计
7. Web 端重构为新的产品叙事与模板入口

### 6.2 一期建议支持

1. OpenClaw 适配
2. Accomplish 适配评估或最小接入
3. 飞书或企微二选一作为首个远程控制渠道
4. 至少 6-12 个高价值模板

### 6.3 一期不要求完成

1. 多 Runtime 全量支持
2. 完整组织级运营后台
3. 复杂的社区互动机制
4. 本地模型自动安装与运行
5. 大规模云端调度系统

## 7. 成本策略

成本控制是本项目的第一原则之一。

### 7.1 成本原则

1. 终端尽量轻
2. Runtime 尽量复用现有生态
3. 模型调用优先共享接入层或低成本 Provider
4. 本地模型仅作为可选项，不作为主方案

### 7.2 当前推荐策略

1. 员工端负责交互、安装和配置
2. 模型侧优先走共享 API 或可替换 Provider
3. 公司只在必要时提供统一接入层，不承诺人手独立服务器
4. 产品价值首先来自“更容易用起来”，而不是“更强的模型性能”

## 8. 目标用户

当前至少服务三类用户：

1. 员工：希望快速拥有可用 AI Agent，不想读文档和折腾命令行
2. 推广者/管理者：希望公司内部 AI 采用更容易推进
3. 比赛评委与业务决策者：希望看到产品有明确定位、可控成本和扩展空间

## 9. 核心用户路径

一期的理想路径应收敛为：

1. 用户打开 Lobster
2. 选择使用场景
3. 选择部署方式
4. 选择目标 Runtime
5. 选择一个模板
6. 由安装桥完成安装与配置
7. 模板写入对应配置文件
8. 用户在桌面或 IM 渠道开始使用

## 10. 仓库重组建议

保留 monorepo，但调整各 app 的角色定义。

### 10.1 `apps/openclaw-installer`

建议重命名方向：后续可演进为 `apps/installer` 或 `apps/lobster-shell`

新的职责：

1. 安装桥主程序
2. 向导流程
3. Runtime 检测与安装
4. 配置写入
5. 模板应用

旧的“仅服务 OpenClaw 且离线内置整个 Runtime”的假设不再作为主前提。

### 10.2 `apps/web`

新的职责：

1. 产品说明页
2. 模板浏览入口
3. 安装路径说明
4. 后续运营能力入口

旧的“社区雷达 + Agent 动态流”降级为非一期主线。

### 10.3 `apps/server`

新的职责建议：

1. 作为模板中心与安装元数据服务的后端雏形
2. 提供模板列表、Runtime 元数据、安装清单
3. 如无必要，不继续优先开发 Agent 社区消息总线

### 10.4 `apps/plugin`

新的职责建议：

1. 从“社区插件”转为“Runtime 接入适配实验区”
2. 用于验证对接 OpenClaw 等 Runtime 的最小集成方式

## 11. 一期实施建议顺序

### Step 1

重置产品文案与代码方向，使团队对项目目标形成统一认知。

### Step 2

清理或降级旧的社区叙事，将安装桥与模板中心提升为主线。

### Step 3

先打通一个最小闭环：

1. 选择 Runtime
2. 选择模板
3. 生成配置
4. 启动成功

### Step 4

再增加渠道接入与 Web 说明层。

## 12. 近期施工原则

从这一版框架开始，后续施工统一遵循下面四条：

1. 先闭环，再扩功能
2. 先解决 adoption，再谈社区协作
3. 先兼容现有生态，再考虑自建体系
4. 先控制成本，再追求技术完备

## 13. 当前结论

Lobster 的当前正确方向，不是继续做一个以 OpenClaw 为中心的社区产品，而是做一个围绕企业 AI Agent 落地的安装桥与模板平台。

这份文档作为后续产品重构、页面改版和代码施工的基线。若后面有新判断，优先更新本文件，而不是让代码实现继续沿用旧叙事漂移。
